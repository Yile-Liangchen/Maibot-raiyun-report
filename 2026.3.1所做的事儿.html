4）问题库

4.1 Docker 镜像拉取失败 已解决

特征：报 unexpected media type text/html。

根因：registry-mirrors 异常返回网页。

解决：删 mirrors，仅保留 127.0.0.1:7890 代理。

4.2 sqlite-web 阻塞初始化 已绕过

这是数据库浏览器，不影响核心聊天链路。

当时为了降低资源占用和避开 manifest 问题，先移除了它。

4.3 EULA / PRIVACY 导致 core 反复退出 已解决

compose 写了 hash 还不够，因为 docker-config/mmc/.env 里的旧值会覆盖。

最终修复：compose 与 mmc/.env 同步到正确值。

4.4 现在最重要的问题：回复慢 待解决

已经不是“能不能用”，而是“为什么体验不丝滑”。

  • 可能是模型端响应慢
  • 可能是规划器阶段较长
  • 可能是代理增加延迟
  • 可能是 4GB 内存下服务抖动

7）访问与 Token

  • 你的服务器不是独立公网 IP，而是 雨云 NAT 映射
  • NapCat WebUI 访问格式：http://公网IP:外网端口/webui/web_login
  • 你的示例：http://103.40.13.93:49619/webui/web_login
  • NapCat 登录 Token：去 NapCat 容器日志里找
  • MaiBot WebUI Token：去 core 日志里找 WebUI Access Token

8）WebSocket 对接

Name : MaiBot-Adapter
URL  : ws://adapters:8095
Token: 留空
Debug: 可开启
格式 : Array
你之前在 shell 里直接输入 ws://maim-bot-adapters:8095 报错是正常的。 这不是终端命令，而是要填在 NapCat WebUI 里的 WebSocket Client 配置框里。

9）回复慢分析

  1. 现在主链路已经通了，所以“慢”不等于“没部署成功”
  2. 更可能是模型阶段慢，而不是 WebSocket 本身慢
  3. 日志里已经出现规划器、思考、动作选择、回复准备等阶段，说明内部链路较长
  4. 若 BaiLian / qwen3.5-plus 接口有排队，也会放大体感延迟
  5. 代理也可能有影响，但在当前证据里不是第一嫌疑人
下一步建议（你后续可继续归档）
  • 记录“我发送消息的时间”
  • 对齐 core 日志：看到消息 → 规划 → 调模型 → 回复生成
  • 对齐 adapters 日志：转发 → 回传
  • 补抓 docker stats，看回复慢时 CPU / 内存是否瞬时抖动

10）这台服务器与官方建议的差异

  1. 访问方式不同：你不是独立公网 IP，而是 NAT 映射访问
  2. 镜像链路不同：你必须依赖本地代理，国内镜像源在你这里反而是坑
  3. 资源更紧：4GB 档位下，每加一个辅助服务都要谨慎
  4. 兼容性更复杂：你实际遇到 platform_name 字段不兼容，不是纯照文档就能无脑过
  5. 更强调日志驱动：你这次几乎每一步都是靠容器日志推进的，这很像真实生产排障
